Shift version: j shifts plus one placement per insertion = (1+1) + (2+1) + (3+1) = 9. (c) No — both O(N2). The saving is a constant factor, which is exactly the kind of thing big-O throws away.

B · Count it

  1. Best: N − 1 — each new element is ≥ the last element of the sorted prefix, so one comparison then break. Worst: N(N−1)/2 — reverse-sorted input forces every insertion to walk the whole prefix.
  2. 1 + 2 + ⋯ + (N−1). The term l is the cost of inserting into a prefix of length l — the worst case of the outer loop's l-th iteration.
  3. NN(N−1)/2N lg Nratio
    828241.17
    1000499,500≈ 9,96650.1
    100× is reached around N ≈ 2,225. The point the N = 8 row is making: at small N the two are nearly identical. Asymptotic separation says nothing about small inputs — which is why real implementations, Timsort included, fall back to insertion sort on short runs.
  4. Minimum n — one list entirely below the other, e.g. [1,2] and [3,4]: every comparison consumes an element of the same list, and the loop exits the moment that list empties. Maximum 2n − 1 — perfectly interleaved, e.g. [1,3] and [2,4]: the loop only exits after all but one element has been appended one at a time.
  5. Each level merges every element exactly once, so work per level is Θ(N) however the level is divided up — one merge of 2n elements costs the same as two merges of n. There are lg N levels, giving N lg N. Counting merge calls alone tells you nothing: there are N−1 of them regardless, and they differ wildly in size. The level view is what makes the units uniform enough to multiply.

C · Prove it

  1. A correct algorithm must be able to output every one of the N! orderings, since each is required for some input. Each leaf is one specific rearrangement, reached by one specific path of comparison outcomes. With fewer than N! leaves, two inputs needing different rearrangements would follow the same path and receive the same rearrangement — so at least one comes out unsorted.
  2. (a) 4! = 24 leaves. (b) A binary tree of height h holds at most 2h leaves; 24 = 16 < 24 ≤ 32 = 25, so minimum height 5. (c) No comparison-based algorithm can sort 4 elements in fewer than 5 comparisons in the worst case. Worth checking against the merge sort tree on slide 19 — its longest paths are exactly 5.
  3. N! = 1·2·⋯·N multiplies N factors each at most N, with at least one strictly less than N for N ≥ 2; NN multiplies N factors all equal to N. So N! < NN.
  4. Let f(k) = k(N+1−k) = −k2 + (N+1)k. This is a downward-opening parabola with roots at k = 0 and k = N+1, so on [1, N] it is smallest at the endpoints: f(1) = 1·N = N and f(N) = N·1 = N, with everything between them larger (peak at k = (N+1)/2). All N factors are therefore ≥ N, so their product N!2 ≥ NN.
  5. f(N) ∈ Ω(g(N)) iff there exist constants c > 0 and n0 such that f(N) ≥ c·g(N) for all N ≥ n0. The constant c absorbs the ½ — take c = ½. Ω is a claim about growth rate, and multiplying by a fixed positive constant doesn't change a growth rate.
  6. If the only thing your algorithm can do with two items is ask which one is bigger, then no amount of cleverness gets you below roughly N lg N steps to sort N items — so merge sort and heapsort are already as fast as that approach can ever be.
  7. No contradiction. The theorem constrains only algorithms whose sole primitive is comparison. Counting sort never compares two elements to each other — it uses a value as an array index, extracting far more than one bit of information per operation. Change the hypothesis, escape the bound. This is the useful takeaway: the theorem tells you where to look for speed — not at better comparisons, but at structure in the values themselves.

D · Verify by running

  1. A1 → 7 comparisons. A3 → 7. B1 best → N−1, worst → N(N−1)/2. B4 → n and 2n−1.
  2. Random input gives insertion sort roughly half the worst case — each insertion walks about halfway into the prefix on average, so ≈ N2/4. Merge sort tracks N lg N closely.
  3. Sorted input of length 1000 → exactly 999 comparisons.
  4. The ratio climbs slowly toward 1: ≈ 0.66 at N = 10, 0.79 at 100, 0.86 at 1000, 0.93 at 106. (Stirling gives lg N! = N lg N − N·lg e + O(lg N).) So lg N! isn't merely at least half of N lg N — asymptotically it is N lg N. The ½ was an artifact of a deliberately crude pairing trick, not a real gap: the proof gave away a factor of 2 in exchange for being three lines long, and Ω didn't care. The theorem is not weakened — a loose constant inside an Ω claim costs nothing.
